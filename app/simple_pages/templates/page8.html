What is SOLID?

SOLID is an acyronym comprising of the five object-oriented design principles. These principles establish the practices for maintaining and extending a project. Despite what the acronym may imply, it is not an ordered list of actions to do. Rather, it is a guideline for good practices in programing to be followed. These principles apply to any object-oriented programming language

S - Single-Responisbility Principle
O - Open-closed Principle
L - Liskov Substitution Principle
I - Interface Segregation Principle
D - Dependency Inversion Principle

SOLID Principles

Single-Responsibility Principle

The Single-Responsibility Principle states:
"A class should have one and only one reason to change"
This principle simply states that a class or function should only have one and only one responsibility. Due to that fact, there should only be one reason to change it. Say you wanted to make a program that finds the average and maximum/largest value in a list. Rather than having a singular function that returns both values, the average and maximum should be returned by two seperate functions that may be called upon. There are many advantages to following this principle.

  • Easier to localize errors
  • More instances of code reusability
  • Easier testing
This is shown in the calculator class as each operation such as add, subtract, divide, and multiply are their own functions.

This example shows how the single responsiblity principle applies to the calculator program. Each function has a single purpose

Open-Closed Principle

The Open-Closed Principle states:
"Objects or entities should be open for extension but closed for modification."
For simplicity sake, this means that you shouldn't need to modify existing code in order to add new functionality. Rather, add what you need and utilize the old code. In the case of the calculator program, we would not need to modify the addition and subtraction functions in order to add the multiplication and division functionality.

It is very likely, if not certain, that when working on a project with multiple developers at once, one will be looking through and adding functionality to existing code that may or may not have been developed by themselves. So rather than going through code that a developer may not understand exactly how it works, other than it just does, they just add on top of the already existing functionality.

This example show how mutliplication and division functionality was added to the calculator program without modifying the pre-existing functions
